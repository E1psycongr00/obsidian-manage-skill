# 단일 소스 규칙

이 스킬의 references/를 본문 작성 규칙의 단일 소스로 사용한다.
노트 타입의 의미 판단과 본문 구조 설계는 여기서 담당한다.
YAML Frontmatter, 필수 필드, 파일 위치, validator 기준은 구조 계약 문서를 따른다.
이 스킬은 본문 의미와 작성 흐름에 집중하며, 구조 계약 자체를 정의하지 않는다.

## 핵심 역할

- 요청의 대상 노트와 작업 유형(신규/확장/리라이트/요약)을 식별한다.
- 노트 유형(CORE/DEEP DIVE/SOLUTION/EVIDENCE/REVIEW/REFERENCE/IDEA)을 확정한다.
- **최소 브리프(필수 요구사항)**로 “작성 설계”를 먼저 만든다.
- 타입을 정한 뒤, 초안 전에 그 타입이 이번 글에서 반드시 남겨야 할 요소를 짧게 고정한다.
- 설계에 따라 본문을 작성하고, 마지막에 Self-check로 품질을 검증한다.
- 사용자가 이미 준 파일, 초안, 제약에서 최대한 추론하고, 진짜 막히는 정보만 1~3개 질문한다.

## 실행 워크플로

`요구사항(Brief) -> 자료 수집(수집/선별/합성) -> Type Plan -> 초안 -> Self-check -> Final`

### 빠른 경로(경량 실행)

아래 조건을 모두 만족하면 전체 워크플로 대신 빠른 경로를 사용한다.

- 기존 노트 1개를 대상으로 한 경미한 편집 요청(문체 통일, 톤 보정, 오탈자/가독성 개선, 짧은 리라이트)
- 노트의 `note-type`이 이미 정해져 있거나, 타입 재판단이 필요 없는 작업
- 새로운 외부 근거 수집이나 구조 변경이 필요 없음

빠른 경로 순서:
- 대상 파일 확인 -> 최소 수정 적용 -> 핵심 패턴 검증(요청 톤/금지 표현) -> 결과 전달
- 이 경우 `자료 수집`, `Type Plan`의 확장 단계를 생략한다.

## 입력 판단 기본값

- 사용자가 대상 파일 경로, 기존 초안, 참고 노트를 줬다면 먼저 읽고 그다음에 필요한 질문만 한다.
- `note-type`이 명시되지 않았더라도 하나의 타입이 뚜렷하면 그 판단을 제안하고 바로 진행한다.
- 사용자가 이미 `CORE로 써줘`, `짧게만`, `이 시리즈 톤으로`처럼 제약을 줬다면 같은 내용을 다시 묻지 않는다.
- 본문 품질보다 방향 탐색이 먼저인 요청으로 보이면, 바로 완성된 본문을 밀어붙이지 말고 목표·독자·핵심 메시지를 짧게 고정한 뒤 작성으로 들어간다.


### 1) 요구사항 수집 (Brief)

#### 3.1) 필수(최소 브리프, 5줄)

아래 5개가 없으면 먼저 추론하되, 불확실하면 질문 1~3개만 던진다.

- 목표(1개): 독자가 무엇을 알게/느끼게/하게 할 건가
- 독자: 누구(초보/동료/미래의 나) + 이미 아는 것/모르는 것
- 핵심 메시지 1문장: “이 글은 결국 ~~다.”
- 형식/분량: (대략) + 매체(Obsidian 노트/블로그/문서 등)
- 금지사항(2~3개): 과장/단정/장황함/공격적 표현/불필요한 전문용어 등

#### 3.2) 선택(있으면 품질 업)

- 범위/비범위(Scope/Non-scope)
- 근거 수준(경험담 중심 vs 자료/수치/인용 필요)
- 톤/문체(“-다”체 등)
- 독자가 바로 할 수 있는 다음 행동(CTA) 필요 여부

> 원칙: “필수”만으로도 작성 가능해야 한다. 선택 항목은 품질을 높이는 옵션이다.

### 2) 노트 타입 결정 (결정 트리)

헷갈릴 때 아래 순서로 예/아니오로 빠르게 확정한다.
이 결정은 글의 의미와 본문 구조를 위한 판단이다.
타입별 frontmatter 이름/필수 여부/파일 위치는 구조 계약 문서를 따른다.

1. 외부 자료를 보존하고 출처와 함께 정리하는가? → REFERENCE
2. 자료/실험/사실을 근거로 결론을 주장하는가? → EVIDENCE
3. 문제 정의~해결 과정을 재사용 가능하게 남기는가? → SOLUTION
4. 보편 개념을 핵심만 압축해 전달하는가? → CORE
5. “왜?”를 파고드는 탐구의 흔적이 핵심인가? → DEEP DIVE
6. 근거 없어도 되는 나의 판단/후기/생각 중심인가? → REVIEW
7. 아직 분류가 애매한 임시 정리/초기 아이디어인가? → IDEA

추가 원칙:
- 사용자가 note-type을 명시했다면 하드 규칙 충돌이 없는 한 우선 존중한다.
- 타입을 바꿔야만 품질이나 규칙을 만족할 수 있으면, 바꾸는 이유를 한 줄로 설명하고 진행한다.

### 3) 자료 수집 (수집 → 선별 → 합성)

목표: “많이 모으기”가 아니라 “좋은 글을 위한 정보 품질”을 확보한다.

#### 3.1 수집(Collect)

- 키워드/관점을 넓게 잡아 후보 자료를 모은다(너무 한 방향으로만 모으지 않음).

#### 3.2 선별(Filter)

- 아래 기준으로 채택 자료 3~7개만 남긴다.
  - 신뢰도(1차 출처/공식 문서/검증된 글)
  - 최신성(주제가 최신에 민감하면 우선)
  - 목표 적합성(이 글의 핵심 메시지에 직접 기여하는가)
- 웹 기반 출처(기사, 논문 landing page/DOI, 공식 문서, 블로그, 보고서 페이지)는 제목만 보고 URL을 추정하지 않는다.
- REFERENCE/EVIDENCE에 쓰일 웹 기반 출처는 최종 반영 전에 실제 브라우저 도구 또는 웹 도구로 해당 URL에 접속해 존재 여부, 문서 제목, 호스트가 맞는지 검증한다.
- 검증에 실패한 URL(404, soft 404, 다른 문서로 리다이렉트, 접근 불가 추정)은 그대로 `source`에 적지 않는다. 가능하면 검증된 정식 URL이나 아카이브 URL을 찾고, 끝내 못 찾으면 URL을 추정 기입하지 않는다.

#### 3.3 합성(Synthesize)

- “요약”이 아니라, 핵심 메시지에 맞게 재구성한다.
- 관점이 충돌하면, 하나로 억지로 합치지 말고 관점 분리로 정리한다.

### 4) Type Plan

필수 로딩:
- `references/rules/note-types.md`
- `references/rules/default-writing-style.md`

- 모듈 단위 맥락이 필요하면 별도 `.context.md`를 찾지 말고 인덱스 노트와 `Map of Content`를 먼저 읽는다.

추가 로딩:
- 유형별 상세 규칙 파일들 (필요시 로딩)
  - CORE: `references/rules/note-type-core.md`
  - DEEP DIVE: `references/rules/note-type-deep-dive.md`
  - SOLUTION: `references/rules/note-type-solution.md`
  - EVIDENCE: `references/rules/note-type-evidence.md`
  - REVIEW: `references/rules/note-type-review.md`
  - REFERENCE: `references/rules/note-type-reference.md`
  - IDEA: `references/rules/note-type-idea.md`

조건부 로딩:
- 기존 노트를 읽고 분석해야 하는 경우 `references/rules/note-read-tip.md`
- 실제 작성에 직접 기여하지 않는 규칙 파일은 한 번에 다 읽지 않는다. 현재 타입과 작업에 필요한 것만 선택한다.

### 4.1) Draft Contract

타입을 확정했으면 초안 전에 아래 4가지를 내부적으로 짧게 잠근다.

- 이 노트가 독자에게 남겨야 할 한 줄
- 이번 타입에서 반드시 살아 있어야 할 구조 요소 2~4개
- 이번 초안에서 과감히 덜어낼 요소
- 마지막에 남길 재사용 문장, 질문, 체크포인트 중 하나

타입별 기본 계약:

- CORE: 정의/핵심 개념으로 시작하고, 본문 안에 `무엇이 아닌가` 또는 `자주 헷갈리는 지점`을 최소 1회 남긴다.
- DEEP DIVE: 시작부터 핵심 질문이나 긴장을 드러내고, 빠른 처방과 실제 메커니즘의 차이를 최소 1회 보여준다.
- SOLUTION: 문제 정의 다음에 선택의 분기점 또는 실패한 접근을 드러내고, 끝에는 재사용 가능한 체크 질문이나 한계선을 남긴다.

### 5) 초안 작성

- 확정한 노트 타입의 최소 구조를 따른다(필요하면 확장).
- 이미 Vault에 있는 내용이면 링크로 연결한다(요구 없으면 과도한 wikilink 금지).
- `개요`, `정리`, `대응 방법` 같은 범용 제목을 습관적으로 복제하지 않는다. 섹션 제목만 봐도 이 노트의 지적 역할이 드러나게 만든다.
- 주제에서 벗어나는 내용은 본문을 흐리지 말고 짧은 콜아웃으로 분리한다.
(콜아웃은 “헤더 내 문단 마지막” 원칙을 따른다.)
- 콜아웃 제목 줄은 기본적으로 `> [!type] Title: ...` 형식으로 쓴다.
- REFERENCE 노트에서 온라인 출처를 다룰 때는 `source`에 실제 검증된 URL을 넣는다. 기사, 논문, 문서 페이지처럼 URL이 출처 식별의 핵심인 경우에는 URL을 생략하지 않는다.
- 책, 오프라인 문헌, 내부 인쇄물처럼 안정적인 웹 원문이 없는 출처만 예외적으로 문헌 식별자(서지 정보 등)로 적고, 없는 URL을 만들어 넣지 않는다.
- 사용자가 `이 노트처럼`, `같은 구성으로`, `이 시리즈 톤에 맞춰`처럼 기존 노트나 형제 노트를 참고 기준으로 지정하면, 초안 작성 직전에 해당 참고 노트를 다시 읽고 수정된 최신 본문을 기준으로 언어, 섹션 리듬, 코드블록 형태를 맞춘다. 같은 세션에서 예전에 읽은 버전만 계속 참고하지 않는다.
- 사용자가 산출물 상세도를 명시하면 그 턴에서는 지정한 상세도까지만 본문을 출력한다. 필요하면 마지막에 확장 가능한 다음 단계만 1줄로 제안한다.
- 설명이 짧아 오해 가능하면 예시/비유/간단한 도식(mermaid/excalidraw/이미지 프롬프트)을 고려한다.
- IDEA는 임시/보류형 본문 구조로 작성한다.
- 실제 파일 배치(`idea/` 디렉토리)와 frontmatter 적용 여부는 구조 계약 문서를 따른다.
- CORE라면 정의 다음에 경계, 오해 방지, 헷갈리는 포인트 중 하나를 짧게라도 넣어 개념의 외곽을 보여준다.
- DEEP DIVE라면 질문 또는 긴장으로 시작하고, 단순 처방과 구조 원인의 차이를 적어도 한 번은 드러낸다.
- SOLUTION이라면 해결 순서만 적지 말고, 어떤 분기에서 왜 그 선택이 나왔는지와 어디까지 통하는지를 짧게 남긴다.
- 마무리는 `한 문장 요약`, `기억할 질문`, `재사용 체크` 중 현재 타입에 맞는 것으로 닫아 독자가 다시 꺼내 쓸 손잡이를 만든다.

### 6) Self-check

아래를 통과하도록 다듬는다.

- 첫 3문장 안에 “무슨 글인지/왜 읽는지”가 드러나는가?
- 섹션 제목만 읽어도 이 노트가 요약인지 탐구인지 해결 기록인지 드러나는가?
- 문단마다 “그래서 뭐?”에 답하는가?
- 주장에는 최소 1개의 근거(예시/데이터/경험)가 붙는가? (타입에 따라 완화 가능)
- 문단 첫 문장만 읽어도 논지가 이어지는가?
- 전문용어는 정의/예시/비유 중 하나가 붙는가?
- (필요한 글이라면) 독자가 바로 할 수 있는 구체적 행동 1개가 있는가?
- 타입별 계약이 실제 본문에도 보이는가?
  - CORE: 경계/오해 방지 포인트
  - DEEP DIVE: 질문, 메커니즘, 대비
  - SOLUTION: 분기점 또는 실패한 접근, 재사용 체크 또는 한계선

### 7) Finalize

- 최종 결과를 사용자 요청 형식에 맞춰 전달한다.
- 추론으로 채운 전제가 있으면 결과 끝에 1~2줄로만 짧게 밝힌다.
- 구조/메타데이터/아카이빙 후속 작업이 필요하면, 그 적용은 이 스킬 범위 밖으로 둔다.

## 완료 기준 (Done Criteria)

아래를 모두 만족하면 작업을 완료로 본다.

- 사용자 요청 범위(대상 파일/요청 유형)만 반영됨
- 요청한 수정 목적(예: 톤 통일, 리라이트, 유형 맞춤 구조)이 결과물에서 확인됨
- 불필요한 구조 변경/메타데이터 변경이 없음
- 후속 구조 작업이 필요하면 범위가 분리되어 있음

## 실패/복구 가이드

- 대상 파일/경로를 찾지 못하면: 추정 생성하지 말고, 확인 가능한 후보 경로만 제시한다.
- 사용자 요구와 기존 규칙이 충돌하면: 요구 우선으로 최소 수정안을 제시하고, 규칙 예외를 한 줄로 명시한다.
- 근거가 필요한 타입인데 자료가 부족하면: 단정 대신 `가정/불확실성`을 표시하고, 추가 근거 요청 또는 저신뢰 초안으로 분리한다.
- REFERENCE용 웹 출처 URL을 검증하지 못하면: 추정 URL을 쓰지 말고, 실제로 열리는 대체 URL(정식 페이지/DOI/아카이브)을 찾거나 웹 검증 실패 사실을 명시한다.

## 태도(운영 원칙)

- 규칙 준수보다 좋은 글과 사용자 요구 충족을 우선한다.
- 다만 근거가 필요한 영역(EVIDENCE/REFERENCE)은 비판적 태도로 선별한다.
- 여러 관점이 하나로 설명될 수 없으면, 억지로 합치지 말고 분리하거나 새 노트로 여러개 다룬다.



## 범위 경계

- 본문 작성, 리라이트, note-type 의미 판단을 다룬다.
- 구조 변경, 메타데이터 스키마 결정, 아카이빙 정책, 모듈 운영 계약은 직접 정의하지 않는다.

## 금지 사항

- 모듈 생성/이동/아카이빙/구조 개편을 직접 처리하지 않는다.
- 메타데이터 스키마를 직접 정의하거나 구조 정책을 결정하지 않는다.
